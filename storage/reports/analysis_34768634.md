### 1. Implement Repository Pattern for Persistence
Currently, the codebase has separate, incompatible implementations for JSON and MongoDB storage, and the UI (sidebar.py) is tightly coupled to the SaveLoadManager. Implementing a common Storage Interface (Repository Pattern) allows the application to switch between local and cloud storage without modifying the business logic or UI layers.

**Current Implementation:**
```python
if st.button("Save Current Game", use_container_width=True):
    save_data = engine.get_full_save_state()
    if SaveLoadManager.save_game(save_data, save_filename):
        st.success(f"Game saved as {save_filename}!")
```

**Suggested Implementation:**
```python
from abc import ABC, abstractmethod

class IStorageProvider(ABC):
    @abstractmethod
    def save(self, identifier: str, data: dict) -> bool:
        pass
    @abstractmethod
    def load(self, identifier: str) -> dict:
        pass

# In sidebar.py
if st.button("Save Current Game", use_container_width=True):
    if engine.storage.save(save_filename, engine.get_full_save_state()):
        st.success("Game saved!")
```

### 2. Replace Global Singletons with Dependency Injection
The codebase relies on global instances (e.g., llm_client, story_generator, settings) imported across modules. This creates tight coupling and makes unit testing nearly impossible because you cannot easily mock the AI client or configuration. Moving to Dependency Injection (DI) improves modularity and testability.

**Current Implementation:**
```python
from ai.llm_client import llm_client

class StoryGenerator:
    def __init__(self):
        self.client = llm_client
```

**Suggested Implementation:**
```python
class StoryGenerator:
    def __init__(self, llm_client: LLMClient, settings: Settings):
        self.client = llm_client
        self.settings = settings

# In app.py
client = LLMClient()
generator = StoryGenerator(llm_client=client, settings=settings)
engine = GameEngine(story_generator=generator)
```

### 3. Introduce Typed Data Transfer Objects (DTOs) for AI Responses
The StoryGenerator returns raw dictionaries parsed from JSON. This is error-prone and relies on string keys (e.g., 'scene_text'), which can lead to KeyErrors if the LLM output deviates slightly. Using a dataclass or Pydantic model for AI responses ensures a strict contract between the AI layer and the Game Engine.

**Current Implementation:**
```python
def generate_next_beat(self, ...) -> dict:
    raw_response = self.client.generate_response(prompt)
    return self._clean_json_response(raw_response)
```

**Suggested Implementation:**
```python
from dataclasses import dataclass

@dataclass
class StoryBeat:
    scene_text: str
    choices: list[str]
    alignment_shift: int

def generate_next_beat(self, ...) -> StoryBeat:
    raw_response = self.client.generate_response(prompt)
    data = self._clean_json_response(raw_response)
    return StoryBeat(**data)
```
